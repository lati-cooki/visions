# Visions — Small Business AI Advisor — CLAUDE.md

## Project Summary

**Visions** is an AI-as-a-service web app that helps small business owners discover and adopt
AI tools and agents. Users complete a 3-step intake (business type → pain points → team
size/budget), verify their email (a 4th step), then receive a personalized AI plan generated by
Claude — which is also emailed to them. The app includes a booking/consultation flow, a local
provider directory, a follow-up chat, and shareable plan links.

**San Diego is the flagship market.** The brand is city-agnostic by design — "San Diego" lives
in config (`src/config/site.js` on the frontend, Worker vars on the backend), not hardcoded —
so additional cities or verticals can be added later without a rewrite.

**Owner:** Troy Latimer / Lati Cooki LLC
**Origin:** Built as a React artifact in Claude.ai (the "SD Small Business AI Advisor"), then
productionized into this full-stack app, rebranded to Visions. The original artifact is archived
at `docs/prototype/sd-biz-ai-advisor.jsx`.

## Current State (full-stack, Phase 1 complete)

A **Vite + React + Tailwind** frontend and a **zero-dependency Cloudflare Worker** backend
backed by **D1**. The Anthropic call is server-side (key held as a Worker secret). Plans persist
to D1 and are shareable at `/plan/:id`; bookings persist too. The frontend keeps a mock mode so
`npm run dev` is fully demoable without a backend or key.

**Live:** deployed to Cloudflare and serving the **l8ti.com** apex (+ `www`) as Custom Domains
on the `visions` Worker. Generation uses the Messages API path (`USE_MANAGED_AGENT="false"`).
Abuse protection is in place: an **email-verification gate** in front of plan generation
(Turnstile-gated `/api/verify/start` emails a 6-digit code; `/api/verify/check` returns a
stateless HMAC token that `/api/plan` requires) plus **daily spend caps** — a per-email cap
(default 3) and a global cap (default 200, ~$10/day) counted from the `plans` table, which is the
real spend ceiling — and per-IP rate limiting. Plans are emailed to the verified address on
generation; booking notifications email the owner (both pending the l8ti.com Email Sending domain
onboarding). Task tracking was removed; the emailed plan is now the "keep it" mechanism. An
**Access-gated admin page** at `/admin` (read-only Bookings/Plans tables + CSV export), gated by
Cloudflare Access at the edge with Worker-side JWT verification as defense-in-depth. Sobriety
Pursuit was relocated to **sp.l8ti.com** (separate `l8ticom` Worker). See
`docs/superpowers/specs|plans/2026-06-15-l8ti-cutover*`.

### Run it

```bash
npm install

# Frontend only, on mock data (no key/backend needed):
npm run dev          # http://localhost:5173

# Backend tests (zero-dependency, node --test):
npm test

# Full stack locally (serves built frontend + API + local D1):
cp .dev.vars.example .dev.vars   # add your ANTHROPIC_API_KEY
npm run db:schema:local          # apply schema to the local D1
npm run db:migrate:local         # apply migrations/ (email-gate) to the local D1
npm run build
npm run worker:dev               # wrangler dev — http://localhost:8787

# Production build (uses .env.production → real API, mock off):
npm run build
```

### Deploy (Cloudflare)

The D1 database is already provisioned (`database_name = "visions"`, id in `wrangler.toml`,
region wnam) and its schema is applied to production. To go live:

```bash
wrangler secret put ANTHROPIC_API_KEY   # one-time, server-side key
npm run db:schema:remote                # idempotent; already applied
npm run deploy                          # vite build + wrangler deploy
```

The Worker serves both `/api/*` and the static frontend (Cloudflare Static Assets with
`run_worker_first = ["/api/*"]`; non-API routes get SPA fallback so `/plan/:id` resolves).

### API contract

| Method + path           | Body                                          | Returns                              |
|-------------------------|-----------------------------------------------|--------------------------------------|
| `GET /api/health`       | —                                             | `{ ok: true }`                       |
| `POST /api/verify/start`| `{ email, turnstileToken }`                   | `{ ok: true }` (Turnstile-gated; emails a 6-digit code; per-email 60s send throttle; `devCode` only when `VERIFY_DEV_ECHO=true`) |
| `POST /api/verify/check`| `{ email, code }`                             | `{ token }` (stateless HMAC token, ~30 min TTL; 5-attempt cap per code) |
| `POST /api/plan`        | profile `{ businessType, painPoints[], teamSize, budget, extraContext }` + `verifyToken` | `{ id, plan }` (verifies the token, enforces per-email + global daily caps, persists the verified email, emails the plan; also persisted) |
| `GET /api/plan/:id`     | —                                             | `{ id, profile, plan, createdAt }`   |
| `POST /api/chat`        | `{ profile, headline, history[], message }`   | `{ reply }`                          |
| `POST /api/booking`     | `{ planId?, name, email, phone, preferred, message }` | `{ ok, id }`                 |
| `GET /api/admin/bookings`| — (`?format=csv` → CSV)                      | `{ bookings: [...] }` (Cloudflare Access-gated; newest-first) |
| `GET /api/admin/plans`  | — (`?format=csv` → CSV)                       | `{ plans: [...] }` (Cloudflare Access-gated; summary fields incl. parsed headline, newest-first) |

The frontend's `src/lib/api.js` is the client for this contract; in mock mode it returns
`src/lib/mockData.js` shaped identically.

## Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| API model | `claude-sonnet-4-6` (env `ANTHROPIC_MODEL`) | Fast/cheap for an interactive consumer flow, strong at structured JSON. Project's documented choice; configurable. |
| Plan format | Structured outputs (`output_config.format` + JSON schema) | Guarantees parseable, well-shaped plans — fixes the prototype's strip-fences-and-hope parsing. |
| Backend | Zero-dependency Worker, raw `fetch` | Troy's "zero-dependency, no framework" standard; no SDK to bundle. |
| Generation path | Managed Agent (sessions) or Messages API, via `USE_MANAGED_AGENT` | Drives `agent_011CZtoh5iVJGPjFmdXkSDzo` in `env_015fkJc7jYAiMT6vN1DMPMBt` per request; both paths use the `ANTHROPIC_API_KEY` secret. Agent path is prompt-enforced JSON + heavier; Messages path keeps schema-guaranteed structured outputs. See `agents/`. |
| Persistence | Cloudflare D1 (`plans` w/ `email`, `bookings`, `email_verifications`; `providers` reserved) | Serverless SQL; enables share/load + lead capture + verification + analytics. |
| Routing | react-router-dom (`/`, `/plan/:id`, `/admin`) | Shareable plan URLs + a load flow + the admin page. |
| Admin auth | Cloudflare Access (Zero Trust) gates `/admin*` + `/api/admin/*` at the edge; the Worker re-verifies the `Cf-Access-Jwt-Assertion` JWT (RS256 via Web Crypto against the team JWKS, checks aud+exp, fails closed; `worker/src/lib/access.js`) | No secrets — Access uses public-key JWTs; edge gate + Worker check is defense-in-depth. Config via Worker vars `ACCESS_TEAM_DOMAIN` + `ACCESS_AUD`; `.dev.vars` `ACCESS_DEV_BYPASS=true` skips it for local `wrangler dev`. |
| Key handling | Worker secret (`ANTHROPIC_API_KEY`) | Never in the client bundle. |
| Config-driven location | `src/config/site.js` + Worker `[vars]` | Keeps "San Diego" out of code for future markets. |
| Email gate + spend caps | Turnstile-gated email verification (stateless HMAC token) + per-email & global daily caps counted from `plans` | Verification is the friction/identity layer; disposable emails weaken it, so the **global daily cap is the real spend guarantee** (~$10/day). |
| Plan delivery | Plan emailed to the verified address on generation (`ctx.waitUntil`, non-fatal) | No client task store — the emailed plan is the "keep it" mechanism; task tracking was removed. |

## Prompt Engineering Notes

Prompt construction lives in `worker/src/lib/prompts.js`:

- **Plan** (`buildPlanRequest`): system prompt sets the advisor role + local awareness from the
  `LOCATION_NAME`/`LOCATION_FLAVOR` Worker vars; the user message carries the profile. Output is
  constrained by `PLAN_SCHEMA` (structured outputs) — shape: `headline`, `quick_wins[]` (title,
  description, tools[], effort ∈ easy|medium|advanced, monthly_cost, task), `agent_opportunity`
  (title, description, impact), `next_step`. The `effort` field maps to UI badges: easy → "Quick
  Start", medium → "Some Setup", advanced → "Deeper Build". The `quick_wins[].task` field is
  **retained** (the task board was removed, but this renders as the per-win "Action:" callout in
  `PlanView` — kept intentionally).
- **Chat** (`buildChatRequest`): carries the profile + plan headline as system context plus the
  prior turns, so answers stay grounded without regenerating the plan.
